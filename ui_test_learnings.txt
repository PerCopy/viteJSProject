2. ExecutionRecorder — CORRECT USAGE
===========================================================================

Constructor signature:  new ExecutionRecorder(acceptanceCriteriaId, testTitle)
  - acceptanceCriteriaId : string  (e.g. "create_event_with_valid_details")
  - testTitle            : string  (e.g. "Create Event Successfully With Valid Inputs")

WRONG — passes Playwright's TestInfo object as second arg:
  const recorder = new ExecutionRecorder("my_id", testInfo);   // TypeError at save()

CORRECT:
  const recorder = new ExecutionRecorder("my_id", "Human-readable test title");

Why it breaks: ExecutionRecorder.save() calls this.testTitle.replace(...).
If testTitle is a TestInfo object (not a string), .replace is undefined → TypeError.
The test assertions may have already passed (CODEVALID_TEST_ASSERTION_OK printed)
before the crash, so the test still fails in Playwright even though the UI worked.

Always call recorder.save(testInfo) at the very end, passing testInfo only there.

3. SELECTOR STRATEGY — WHAT WORKS IN THIS APP
===========================================================================

DO USE:
  page.locator('input[name="fieldName"]')   — reliable; name attrs are always present
  page.getByRole("button", { name: /text/i })
  page.getByRole("heading", { name: "..." })
  page.getByPlaceholder("...")              — safe for text inputs and textareas

DO NOT USE:
  page.getByLabel("Start Date *")   — FAILS in this repo
  page.getByLabel("End Date *")     — FAILS in this repo

Why getByLabel fails here:
  src/components/ui/Input.jsx renders <label> and <input> as siblings inside a
  <div>, with NO htmlFor/id pairing and NO aria-labelledby/aria-label.
  Playwright's getByLabel() requires an ARIA association to resolve the input.
  Without it the locator never matches and the test times out (30 s).

For date inputs specifically use:
  page.locator('input[name="startDate"]')
  page.locator('input[name="endDate"]')

Verify name attributes by reading src/pages/Events.jsx before writing selectors.

5. API MOCKING PATTERNS
===========================================================================

All API calls go through src/utils/api.js using fetch() with no base URL
(proxied by Vite). Intercept with page.route("**/api/...").

Confirmed API paths (read from src/utils/api.js):
  GET  /api/events          — fetch events list
  POST /api/events          — create event; returns created event object
  POST /api/auth/signin     — sign in
  POST /api/auth/signup     — sign up
  GET  /api/registrations/:eventId
  POST /api/registrations

mockEventScenario(page, { initialEvents, createdEvent }) in helpers/mock-api.js:
  - Single page.route("**/api/events") handler covers both GET and POST.
  - Tracks a local `eventCreated` flag; GET returns initialEvents before POST,
    returns [createdEvent] after POST (covers page reload / re-navigation).
  - POST returns HTTP 201 with the createdEvent object.

Always include registrationCount: 0 in event fixture objects — the app
renders event.registrationCount and will show "undefined registered" if missing.

Event object shape expected by Events.jsx:
  { id, title, description, location, startDate, endDate, registrationCount }

IMPORTANT — intercepting POST but not fulfilling it:
  If a page.route() handler matches a POST but returns without calling
  route.fulfill() or route.continue(), the request hangs indefinitely and the
  test times out. Always handle every method branch explicitly or call
  route.continue() as a fallback.
  Example of a valid pattern for a test that only needs GET:
    page.route("**/api/events", async (route) => {
      if (route.request().method() === "GET") {
        await route.fulfill({ status: 200, body: JSON.stringify([]) });
        return;
      }
      await route.continue();   // let POST through to real backend or fall-through
    });

8. APP_BUG vs TEST_DEFECT — WHEN NOT TO EDIT THE TEST
===========================================================================

Some tests are written intentionally to verify missing application behaviour.
These tests CANNOT be fixed within .codevalid/ui/ and must be reported as
APP_BUG. Indicators:

- The test's mock has a block intentionally commented out with a note like:
    // NOTE: Must follow, Do not uncomment this we want to reproduce this bug
  This means the test is designed to exercise real app logic, not a mock.
  Do NOT uncomment the block or add a substitute mock to make the test pass.

- The acceptance criteria describe behaviour (e.g. "start date must be before
  end date") that does not exist anywhere in the app source.

- The error message the test expects (e.g. getByText("Start date must be
  before end date")) is never rendered by any component.

Diagnosis checklist before editing a test:
  1. Search src/ for the exact error string the test expects.
  2. Confirm it is rendered in a React component (not just in a comment/test).
  3. If absent, identify which app file would need to produce the string.
  4. If that file is outside .codevalid/ui/, the outcome is APP_BUG — report
     it and make no file changes.

Known app validation gap (as of 2026-07-22):
  src/pages/Events.jsx validateForm() (lines 51–61) checks only field presence.
  It does NOT compare startDate vs endDate. If a test expects the message
  "Start date must be before end date" from client-side validation, it will
  fail until the app adds:
    if (formData.startDate && formData.endDate &&
        formData.startDate >= formData.endDate) {
      newErrors.startDate = "Start date must be before end date";
    }
  backend/server.js POST /api/events (lines 98–116) also has no date-order
  guard. Both locations need the fix; neither is in .codevalid/ui/.

9. AVOID / DO NOT LIST
===========================================================================

- Do NOT use page.getByLabel() for any input in this app until Input.jsx
  is updated to include htmlFor/id ARIA linkage. Use input[name] instead.

- Do NOT pass the Playwright TestInfo object as the second arg to
  ExecutionRecorder. Always pass a plain string title.

- Do NOT import helpers using ../helpers/ from inside a task subfolder.
  The correct depth is ../../helpers/.

- Do NOT skip seedAuthenticatedSession(page) for any test that navigates to
  /events or any other protected route.

- Do NOT call page.goto() before setting up page.route() interceptors and
  page.addInitScript(). Routes and init scripts must be registered first.

- Do NOT hardcode localhost URLs in page.goto(). Use relative paths so
  baseURL from playwright.config.js applies.

- Do NOT import from .codevalid/ui/mock/mock-data.js or mock-server.js in
  spec files. Those are server-side files. Use helpers/mock-api.js instead.

- Do NOT guess API route names. Always verify in src/utils/api.js before
  writing a page.route() intercept.

- Do NOT omit registrationCount from event fixture objects — the UI renders it.

- Do NOT add a mock fulfillment to a test that intentionally has one commented
  out with a "do not uncomment" note. That is an APP_BUG marker, not a gap
  to fill with a workaround.

- Do NOT assume a page.route() handler is complete if it lacks a branch for
  every HTTP method. An unhandled POST/PUT/DELETE will hang the request and
  cause a timeout. Always add route.continue() as a fallback.

- Do NOT search only the test file when a test assertion fails. Always cross-
  check src/ to confirm the expected text/element actually exists in the app.
